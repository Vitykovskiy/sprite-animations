# 02 — Business Requirements

> Заполняется агентом после качественного снятия задачи.

---

## Бизнес-требования

| ID     | Требование                  | Приоритет  | Источник   |
|--------|-----------------------------|------------|------------|
| BR-01  | Команда должна иметь собственную библиотеку для sprite-анимаций на `canvas` | High | Пользователь |
| BR-02  | Решение должно быть пригодно для внутреннего повторного использования как npm-пакет | High | Пользователь |
| BR-03  | Настройка ассетов и параметров должна быть доступна без отдельного production UI | Medium | Пользователь |

---

## Функциональные требования

| ID     | Требование                  | Приоритет  | Связанные BR |
|--------|-----------------------------|------------|--------------|
| FR-01  | Библиотека должна загружать sprite sheet как одно изображение | High | BR-01, BR-02 |
| FR-01A | Библиотека должна загружать frame sequence как массив отдельных кадров | High | BR-01, BR-02 |
| FR-02  | Библиотека должна поддерживать регулярную сетку кадров по параметрам `frameWidth`, `frameHeight`, `columns`, `rows` | High | BR-01 |
| FR-03  | Библиотека должна воспроизводить анимацию с управлением `play`, `pause`, `stop`, `loop` | High | BR-01 |
| FR-04  | Библиотека должна поддерживать тайминг через `fps` и/или `duration` | High | BR-01 |
| FR-05  | Если заданы и `fps`, и `duration`, приоритет у `duration`, а `fps` ограничивает частоту обновления и допускает frame skipping | High | BR-01 |
| FR-06  | Библиотека должна позволять масштабировать спрайт и задавать позицию на `canvas` | High | BR-01 |
| FR-07  | Библиотека должна позволять изменять позицию анимации во времени через runtime API | Medium | BR-01 |
| FR-08  | Проект должен включать dev playground для настройки и проверки параметров анимации | High | BR-03 |
| FR-09  | Playground должен позволять сохранять конфиг анимации в JSON | Medium | BR-03 |

---

## Нефункциональные требования

| Категория       | Требование                  | Метрика              |
|-----------------|-----------------------------|----------------------|
| Производительность | Runtime не должен требовать WebGL для базового сценария | Стабильная работа в `CanvasRenderingContext2D` |
| Безопасность    | Секреты и токены не хранятся в репозитории | `.env` не коммитится |
| Надежность      | Поведение тайминга и frame skipping должно быть детерминированным | Одинаковый набор входных параметров дает одинаковый результат |
| Масштабируемость| Архитектура должна позволять позже добавить WebGL и adapters без переписывания core | Core API и renderer разделены |

## Acceptance Criteria

### FR-01: Загрузка sprite sheet
- [ ] Можно загрузить изображение sprite sheet по URL или объекту `HTMLImageElement`.

### FR-01A: Загрузка frame sequence
- [ ] Можно загрузить массив URL или набор `HTMLImageElement` / `ImageBitmap` кадров.
- [ ] Последовательность кадров может быть передана в `AnimationPlayer` через общее `totalFrames`.

### FR-02: Регулярная сетка кадров
- [ ] Пользователь задает `frameWidth`, `frameHeight`, `columns`, `rows`.
- [ ] Библиотека корректно вычисляет координаты кадров по индексу.

### FR-04: Тайминг анимации
- [ ] При задании только `fps` анимация идет покадрово.
- [ ] При задании только `duration` кадры распределяются по времени.
- [ ] При одновременном задании `fps` и `duration` используется приоритет `duration`.

### FR-08: Playground
- [ ] Playground позволяет загрузить ассет, задать сетку кадров, позицию, масштаб и тайминг.
- [ ] Playground позволяет сохранить конфиг в JSON.

## Допущения

- Основные форматы входных данных в MVP — регулярный sprite sheet по сетке и frame sequence из отдельных кадров.
- Playground является вспомогательным dev-инструментом.
- Публикация планируется как npm-пакет для браузерного использования.
