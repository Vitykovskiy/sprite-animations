### ADR-001: Использование шаблона AI-dev-template

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Проект инициализирован из шаблона AI-dev-template.

**Решение:**
Использовать структуру и workflow из AI-dev-template как базовую.

**Причина:**
Обеспечить консистентный, предсказуемый процесс разработки с AI-агентом.

**Альтернативы:**
Создание структуры с нуля.

**Последствия:**

### ADR-002: Обязательный commit после каждой завершенной задачи

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Проект ведется через GitHub Issues и GitHub Project, но без явного правила коммитов история изменений могла бы получиться неоднородной и слабо связанной с конкретными задачами.

**Решение:**
После закрытия каждой задачи агент обязан создавать отдельный git commit. Commit message должен включать номер задачи, ее краткое описание и краткое описание выполненной работы. Commit выполняется после обновления кода, документации, issue и статуса карточки в GitHub Project.

**Причина:**
Это делает историю изменений трассируемой, связывает код с конкретными задачами и соответствует ожидаемым best practices разработки.

**Альтернативы:**
Не фиксировать правило коммитов явно; делать произвольные или агрегированные commits на несколько задач.

**Последствия:**
История git становится более подробной и дисциплинированной. Агент не должен завершать задачу без отдельного commit, если пользователь не согласовал исключение.

### ADR-003: Regular grid sprite sheet как формат ассетов MVP

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Для первой версии нужно простое и контролируемое решение для внутренних canvas-анимаций. Рассматривался `json atlas`, но он оказался лишним для стартового use case и плохо совпал с ожиданиями пользователя.

**Решение:**
В MVP используется regular grid sprite sheet с параметрами `frameWidth`, `frameHeight`, `columns`, `rows`. Поддержка atlas и отдельных файлов кадров откладывается.

**Причина:**
Такой формат проще для понимания, настройки и реализации в первой версии, а также лучше подходит для playground.

**Альтернативы:**
`json atlas`; набор отдельных файлов кадров.

**Последствия:**
Первая версия быстрее в реализации, но работает только с равномерной сеткой кадров. Поддержка более сложных схем ассетов переносится в последующие итерации.

### ADR-004: Timing policy с приоритетом duration над fps

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Пользователю нужны сценарии, в которых одновременно заданы `fps` и `duration`, включая длинные анимации, где показываются не все кадры.

**Решение:**
Если задан только `fps`, анимация идет покадрово. Если задана только `duration`, кадры распределяются по времени. Если заданы оба параметра, приоритет у `duration`, а `fps` ограничивает частоту обновления, поэтому допускается frame skipping.

**Причина:**
Это покрывает реальные use cases без усложнения API отдельными режимами проигрывания.

**Альтернативы:**
Разрешить только один из параметров; игнорировать `fps`; всегда требовать показ всех кадров.

**Последствия:**
Нужно явно документировать поведение тайминга и покрыть его тестами, чтобы избежать неоднозначной интерпретации.

### ADR-005: Корневой TypeScript-пакет с разделением на core, renderers и types

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Следующий этап реализации требует уже не только описанной архитектуры, но и стабильной физической структуры пакета, чтобы последующие задачи по asset loading, runtime и playground не спорили о базовых entry points и публичных контрактах.

**Решение:**
Проект стартует как один корневой TypeScript-пакет с исходниками в `src/`. Публичные entry points разделены на корневой экспорт, `./core`, `./renderers` и `./types`. Внутри первой версии core отделен от renderer-слоя на уровне файловой структуры и API.

**Причина:**
Это самый простой путь для MVP: одна кодовая база без монорепозитория, но уже с явной границей между моделью sprite sheet, playback state и renderer-специфичной логикой.

**Альтернативы:**
Монорепозиторий с отдельными пакетами; один плоский `src/` без разделения на subpath exports; привязка renderer-кода к core API.

**Последствия:**
Следующие задачи могут реализовывать runtime и playground, не пересматривая package topology. Если позже появятся adapters или WebGL, их можно будет добавлять как новые subpath exports или отдельные пакеты без слома базового API.

### ADR-006: Browser-first asset loader для sprite sheet

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Core библиотеке нужен отдельный путь загрузки sprite sheet, чтобы runtime не зависел от внешнего кода потребителя в базовом сценарии. При этом загрузчик не должен быть привязан к atlas, фреймворкам или сторонним networking-абстракциям.

**Решение:**
В core добавлен browser-first asset loader `loadSpriteSheetImage(...)`, который принимает URL, `HTMLImageElement` или `ImageBitmap` и возвращает нормализованный загруженный asset с размерами.

**Причина:**
Это минимальный и достаточно гибкий контракт для MVP: он закрывает базовый сценарий загрузки изображения и не мешает позже добавить более сложный loading pipeline.

**Альтернативы:**
Не включать loader в core; использовать только URL-строки; сразу строить общий network loading manager.

**Последствия:**
Core уже умеет принимать и нормализовать sprite sheet input, но пока ограничен browser runtime и не решает более широкие задачи asset orchestration.

### ADR-007: Root-level Vite playground для настройки и проверки runtime

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
MVP требует отдельный dev playground для разработчика и дизайнера. Он должен работать с локальным sprite sheet, показывать live preview и сохранять JSON-конфиг, но при этом не тянуть в проект лишний UI-фреймворк.

**Решение:**
Playground реализован как root-level `index.html` + TypeScript-модуль в `src/playground/`, который обслуживается через Vite. Playground использует существующий core runtime напрямую и не вводит отдельный adapter-слой.

**Причина:**
Такой вариант минимален по сложности, быстро запускается локально и позволяет проверять библиотечный runtime на реальном canvas без лишней архитектуры.

**Альтернативы:**
Отдельное приложение в подпапке; React/Vue playground; отсутствие playground до более позднего этапа.

**Последствия:**
В репозитории появляется root `index.html` как dev entry point. Playground остается частью внутреннего workflow и не меняет production API пакета.

### ADR-008: Раздельные output directories для library build и playground build

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
После добавления playground выяснилось, что library build и Vite build не должны писать в один и тот же `dist/`, иначе quality gate и `npm pack --dry-run` начинают конфликтовать друг с другом.

**Решение:**
Library build продолжает собираться в `dist/`, а playground build выводится в `playground-dist/`.

**Причина:**
Так package artifacts остаются чистыми и пригодными для публикации, а playground сохраняет собственный build output без риска повредить library bundle.

**Альтернативы:**
Один общий `dist/`; отдельный репозиторий или подпакет для playground.

**Последствия:**
`npm pack --dry-run` отражает только библиотеку, а dev playground можно собирать и проверять независимо.

### ADR-009: Канонический JSON-контракт UI/UX элементов для дизайнера

**Дата:** 2026-03-11
**Статус:** Accepted

**Контекст:**
Playground уже содержит заметный набор UI-элементов, а дизайнеру нужен стабильный и машиночитаемый документ, описывающий текущую структуру интерфейса, элементы управления и их поведение.

**Решение:**
Добавлен `docs/11-ui-ux-contract.json` как канонический JSON-документ с описанием layout, секций, control IDs, labels, поведения и responsive-правил. Workflow дополнен обязательным правилом обновлять этот документ при любом изменении интерфейса.

**Причина:**
JSON-формат удобен для передачи дизайнеру, автоматической сверки и поддержки синхронизации между кодом и UI-описанием.

**Альтернативы:**
Хранить описание только в Markdown; не иметь отдельного UI-контракта; описывать интерфейс устно или только через код.

**Последствия:**
Любая UI-задача теперь требует синхронизации кода и `docs/11-ui-ux-contract.json`, иначе документация будет считаться неполной.

### ADR-010: Vue 3 как UI-слой dev playground

**Дата:** 2026-03-13
**Статус:** Accepted

**Контекст:**
Playground вырос из небольшого DOM-скрипта в отдельное приложение с реактивным состоянием, несколькими режимами ассетов, canvas preview и экспортом конфигурации. Imperative TypeScript-код стал хуже масштабироваться и усложнил сопровождение UI.

**Решение:**
Перевести playground на Vue 3 с Composition API через `script setup`. Canvas/runtime orchestration вынести в composable, а public runtime API библиотеки оставить framework-agnostic и без Vue-зависимостей.

**Причина:**
Vue 3 дает более прозрачную модель состояния для playground, снижает связность ручной DOM-логики и улучшает сопровождаемость без необходимости менять архитектуру самой библиотеки.

**Альтернативы:**
Оставить imperative DOM-подход; переписать playground на React; вынести playground в отдельное приложение/репозиторий.

**Последствия:**
В devDependencies появляются `vue`, `@vitejs/plugin-vue` и `vue-tsc`. Typecheck playground теперь идет через `vue-tsc`, а сам playground остается отдельным consumer-слоем поверх существующего core runtime.

### ADR-011: Отдельный frontend style guide для Vue/playground

**Дата:** 2026-03-13
**Статус:** Accepted

**Контекст:**
После миграции playground на Vue 3 best practices были частично зафиксированы в `docs/04-tech-stack.md`, но явного canonical документа по frontend codestyle не было.

**Решение:**
Добавить `docs/frontend-styleguide.md` как отдельный документ с правилами структуры компонентов, composables, TypeScript, template bindings и side effects для frontend-слоя проекта.

**Причина:**
Так codestyle и structural rules становятся явной частью документации проекта, а не зависят от временного контекста диалога.

**Альтернативы:**
Оставить правила только в `docs/04-tech-stack.md`; не фиксировать codestyle отдельно; описывать правила устно в задачах.

**Последствия:**
Все новые frontend-задачи должны сверяться с `docs/frontend-styleguide.md`, а изменения этих правил теперь требуют обновления этого документа и связанных ссылок в workflow/overview.

### ADR-012: Frontend rules split into machine configs and focused docs

**Дата:** 2026-03-13
**Статус:** Accepted

**Контекст:**
Один большой `docs/frontend-styleguide.md` оказался слишком широким: часть правил должна быть машино-проверяемой, а часть лучше держать в коротких документах по слоям.

**Решение:**
Сделать каноничными источниками frontend-правил `eslint.config.js`, `.prettierrc.json`, `stylelint.config.js` и `tsconfig.json`, а human-level guidance разнести по документам `docs/frontend-architecture.md`, `docs/vue-guidelines.md`, `docs/typescript-guidelines.md`, `docs/styling-guidelines.md`. Для агента добавить `docs/frontend.instructions.md`.

**Причина:**
Так форматирование, lint rules и strictness проверяются автоматически, а архитектурные и operational правила остаются короткими и проще поддерживаются.

**Альтернативы:**
Оставить один большой markdown guide; не вводить machine-enforced frontend rules; хранить operational rules только в `AGENTS.md`.

**Последствия:**
Frontend rules теперь меняются в двух слоях: конфиги для машины и короткие документы для людей/агента. При изменении этих правил нужно синхронизировать оба уровня.

### ADR-013: Background preview в playground не входит в JSON-конфиг

**Дата:** 2026-03-15
**Статус:** Accepted

**Контекст:**
Для playground понадобился дополнительный слой visual QA: проверка спрайта на разных типах задника, включая цветовые палитры и локальную background-картинку. При этом existing JSON-конфиг описывает именно runtime-настройки анимации, а не временное окружение preview.

**Решение:**
Добавить background preview как отдельное dev-only состояние playground. Поддерживаются preset-палитры, локальная background-картинка и selectable режимы отображения (`cover`, `contain`, `repeat`, `stretch`). Эти данные не включаются в `PlaygroundConfig`, не экспортируются и не копируются в JSON.

**Причина:**
Background нужен только для визуальной проверки качества ассета. Включение его в JSON-конфиг смешало бы runtime-настройки анимации с временным preview-контекстом и сделало бы конфиг менее каноничным.

**Альтернативы:**
Хранить background в JSON как поля конфига playground; сериализовать картинку как data URL; ограничиться только palette presets без загрузки изображения.

**Последствия:**
Playground получает дополнительный UI-state, но канонический JSON-конфиг остается чистым и относится только к анимации. Background сохраняется только в рамках текущей сессии.
