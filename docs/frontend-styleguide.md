# Frontend Style Guide

Правила кодстайла и структуры frontend-кода для playground и любых будущих UI-слоев проекта.

---

## Область действия

- `src/playground/**/*.vue`
- `src/playground/**/*.ts`
- frontend-конфигурация сборки (`vite.config.ts`, frontend-часть `tsconfig`)

Эти правила не меняют public API библиотеки и не относятся к framework-agnostic core в `src/core` и `src/renderers`.

---

## Базовые принципы

- Playground остается consumer-слоем над библиотекой, а не местом для переноса бизнес-логики из core.
- Код должен быть предсказуемым, типизированным и простым для локальной правки без поиска по нескольким абстракциям.
- Style guide важнее личных предпочтений в оформлении.

---

## Vue Component Style

- Использовать `script setup` во всех новых `.vue` компонентах.
- Один компонент должен отвечать за один UI-блок или экран.
- Не держать сложную runtime-логику прямо в шаблоне.
- Не использовать Options API в новом коде без отдельного обоснования.
- Не использовать class-based Vue patterns.

---

## Composition API Rules

- Использовать `ref` для примитивов и nullable DOM refs.
- Использовать `reactive` для компактных связанных групп состояния формы.
- Использовать `computed` только для производных значений, а не как замену обычным функциям.
- Использовать `watch` только для side effects и синхронизации с внешним миром.
- Не складывать unrelated state в один большой `reactive` объект.

---

## Composable Rules

- Сложная UI-логика, canvas orchestration, asset loading и state transitions выносятся в composable.
- Composable должен описывать поведение, а компонент должен в основном описывать разметку.
- Composable не должен тянуть в себя лишние DOM query через `document.getElementById`.
- DOM-элементы передаются через `ref`.
- Composable не должен импортировать ничего из `src/playground/*.vue`.

---

## Template Rules

- В шаблоне не писать нетривиальные вычисления inline.
- Не вызывать функции с побочными эффектами из template bindings.
- Для классов использовать короткие и очевидные биндинги.
- Повторяющиеся блоки выносить в отдельный компонент только когда это реально уменьшает сложность.
- Если блок используется один раз и короткий, оставлять его в текущем компоненте.

---

## Naming Rules

- Компоненты: `PascalCase.vue`
- Composables: `useXxx.ts`
- Локальные переменные и функции: `camelCase`
- Типы и интерфейсы: `PascalCase`
- Булевы значения: с префиксами `is`, `has`, `can`, `should`, где это уместно

---

## TypeScript Rules

- Не использовать `any`.
- Для public shape данных объявлять явные типы или интерфейсы.
- Для optional form values учитывать оба случая: `string` и `number`, если значение приходит из `v-model`.
- Ошибки runtime обрабатывать через `Error` с понятным сообщением.
- Не прятать небезопасные приведения типов без причины.

---

## File Structure Rules

- Vue entrypoint: минимальный bootstrap в `main.ts`
- Экран/контейнер: `.vue` компонент
- Поведение и orchestration: composable
- Общие UI-утилиты: отдельные `.ts` файлы при появлении повторного использования

Предпочтительная структура:

```text
src/playground/
  App.vue
  main.ts
  usePlaygroundRuntime.ts
  styles.css
```

---

## Styling Rules

- Один frontend surface проекта может использовать общий `styles.css`, пока объем стилей остается обозримым.
- Токены цветов, радиусов и теней хранить в CSS custom properties.
- Не смешивать layout fixes и component skinning без необходимости.
- Новые CSS-классы называть по смыслу, а не по визуальному эффекту.
- Не завязывать CSS на autogenerated Vue attributes.

---

## Side Effects and Lifecycle

- Side effects запускать в `onMounted`, очищать в `onBeforeUnmount`.
- `requestAnimationFrame`, object URLs и другие ресурсы обязаны освобождаться явно.
- Не создавать бесконтрольные циклы обновления через `watch`.

---

## What To Avoid

- Imperative DOM manipulation вместо Vue bindings
- Дублирование runtime-логики библиотеки внутри playground
- Скрытые зависимости между компонентом и глобальным DOM
- Смешение package API, UI state и canvas drawing в одном файле, если файл уже стал трудно читаемым
- “Умные” абстракции без повторного использования

---

## Review Checklist

- Компонент в основном декларативный
- Сложная логика вынесена из `.vue` в composable или utility
- Нет `any`
- Нет прямых DOM query без необходимости
- Vue-слой не протекает в public runtime API
- Side effects имеют cleanup
- Имена соответствуют правилам документа
